4  Random Correlations

4.1 What are Random Correlations?

In the last section, we saw how we can simulate not only random variability between participants’ overall response times, but also variability in the magnitude of slopes. We did this by modelling both \(P_{0p}\) (random intercepts) and \(P_{1p}\) (random slopes):

\[logRT_{ip} = \beta_0 + I_{0i} + P_{0p} + (\beta_1 + P_{1p}) Zipf_{ip} + e_{ip}\]

We simulated the random effects by drawing from independent normal distributions. However, this assumes that random intercepts and slopes are uncorrelated. It’s not unreasonable to expect that there might actually be a correlation here. For example, people who respond more quickly may also show a larger effect of word frequency.

We can simulate these relationships by drawing \(P_{0p}\) and \(P_{1p}\) from multivariate normal distributions:

\[ \begin{pmatrix} P_{0p} \\ P_{1p} \end{pmatrix} \sim \mathcal{N}\!\left(\mathbf{0},\, \Sigma_P\right) \\ \]

Rather than a single standard deviation, as we’ve seen so far, \(\Sigma_P\) is a covariance matrix, describing the standard deviations (variances) of \(P_0\) and \(P_1\) on the diagonal, and the covariances on the upper and lower triangle.

\[ \Sigma_P = \begin{pmatrix} \sigma^2_{P0} & \sigma_{P01} \\ \sigma_{P01} & \sigma^2_{P1} \end{pmatrix} \]

Another way of writing the covariances is as correlations (\(\rho\)) multiplied by the standard deviations \(\sigma_{P0}\) and \(\sigma_{P1}\):

\[ \Sigma_P = \begin{pmatrix} \sigma^2_{P0} & \rho_P \sigma_{P0} \sigma_{P1} \\ \rho_P \sigma_{P0} \sigma_{P1} & \sigma^2_{P1} \end{pmatrix} \]

We will continue to simulate the item random intercepts and the residuals as just independent normal distributions:

\[\begin{aligned} I_{0i} &\sim \mathcal{N}(0, \sigma_{I0}) \\ e_{ip} &\sim \mathcal{N}(0, \sigma_e) \end{aligned}\]

4.3 Simulating Multivariate Normals

To simulate multivariate normal distributions, we can no longer use rnorm(). Instead, we can use the rnorm_multi() function from faux (DeBruine 2020).

For example, to simulate observations from two normally distributed variables, a and b, with a correlation of 0.76, we can do the following:

# number of observations
n_ab <- 250

# means of the normal distributions
mu_a <- 7.2
mu_b <- 0

# SDs of the normal distributions
sigma_a <- 0.5
sigma_b <- 10

# correlation of the normal distributions
r_ab <- 0.76

# simulate observations
ab_df <- rnorm_multi(
  # number of observations
  n = n_ab,
  # number of variables
  vars = 2,
  varnames = c("a", "b"),
  # means
  mu = c(mu_a, mu_b),
  # SDs
  sd = c(sigma_a, sigma_b),
  # correlation
  r = r_ab
) |>
  as_tibble()

print(ab_df)

Above, we passed a single correlation to the r argument. In cases where you have more than 2 variables, you can pass a full correlation matrix.

Show example 
rnorm_multi(
  # number of observations
  n = 1000,
  # number of variables
  vars = 3,
  # means
  mu = c(0, 10, 4.2),
  # SDs
  sd = c(1, 7, 2.5),
  # correlation
  r = matrix(data=c(
    1, 0.5, 0.21,
    0.5, 1, 0.7,
    0.21, 0.7, 1
  ), nrow=3)
)

4.4 Defining Parameters

These are the parameters that I suggest for this simulation. These are the same as before, except now we also have the correlation between s_P0 and s_P1, which I’ve called r_P. I’ve set this to -.74, meaning that participants with larger (slower) average response times should also have a reduced effect of word frequency.

# Fixed-effect intercept and slope
# (same as the parameters in the section on linear models)
b_0 <- 6.85
b_1 <- -0.1

# SD of per-item random intercepts
s_I0 <- 0.08

# SD of per-participant random intercepts
s_P0 <- 0.16
s_P1 <- 0.04
r_P  <- -0.74

# residual SD
s_e <- 0.33

4.5 Simulating Correlated Random Effects

Using the rnorm_multi() example above, write code to simulate the random intercepts and slopes for both items and participants. To ensure that the lmer() model will have enough information to estimate the random correlations, I also recommend increasing n_participants.

Show solution 
# simulate 50 items
n_items <- 50

items <- tibble(
  # item IDs
  i_id = 1:n_items,
  # item word frequencies
  zipf = runif(n_items, 1.5, 7.5),
  # item intercepts
  I_0i = rnorm(n_items, 0, s_I0)
)

# simulate 30 participants
n_participants <- 30

participants <- rnorm_multi(
  n = n_participants,
  vars = 2,
  varnames = c("P_0p", "P_1p"),
  mu = c(0, 0),
  sd = c(s_P0, s_P1),
  r = r_P
) |>
  mutate(p_id = 1:n_participants)

4.6 Visualising Random Effects

If we now visualise the variability between items and participants, you can see that participants with faster response times also tend to have steeper slopes in the effect of Zipf:

ggplot(items) +

  geom_abline(intercept=b_0, slope=b_1, colour="red") +

  geom_abline(intercept=b_0, slope=b_1, colour="red") +
  lims(x=c(1.5, 7.5), y=c(5, 7.5)) +

# (c)
ggplot(participants, aes(b_0 + P_0p, b_1 + P_1p)) +
  geom_point() +
  geom_smooth(method="lm")

As before, the red line shows the fixed effect, while the black lines show variability between (a) items and (b) participants. The blue line in the last panel shows the relationship between per-participant intercepts (\(\beta_0 + P_{0p}\)) and slopes (\(\beta_0 + P_{1p}\)).

4.7 Simulating Trials and Responses

These sections are actually identical to what we did before. We get all combinations of items and participants, and then simulate:

  • \(e_{ip}\) (e_ip) as random noise of mean 0 and SD \(\sigma_e\) (s_e)
  • \(logRT\) (log_rt) using the equation at the top
# generate trials
trials <- crossing(items, participants) |>
  mutate(
    e_ip = rnorm(n(), 0, s_e),
    log_rt = b_0 + I_0i + P_0p + (b_1 + P_1p) * zipf + e_ip
  )

print(trials)

4.8 Parameter Retrieval

To fit a model that can estimate the simulated random correlation, we can now use one single bar (|) rather than double bars (||). This tells lme4 to estimate and account for the influence of random correlations:

m <- lmer(
  log_rt ~ 1 + zipf +
    (1 | i_id) +
    (1 + zipf | p_id),
  data = trials
)
Warning in checkConv(attr(opt, "derivs"), opt$par, ctrl = control$checkConv, : Model failed to converge with max|grad| = 0.00334773 (tol = 0.002, component 1)
  See ?lme4::convergence and ?lme4::troubleshooting.

Oh no — Looks like our model didn’t converge! Convergence warnings occur fairly often for mixed-effects models. They are much more likely to occur when the model structure is complicated (e.g., including random correlations).

Note

Something that will be useful later is that we can also extract messages about nonconvergence like so:

m@optinfo$conv$lme4$messages
[1] "Model failed to converge with max|grad| = 0.00334773 (tol = 0.002, component 1)\n  See ?lme4::convergence and ?lme4::troubleshooting."

This allows us to detect nonconvergence programmatically, as if the model converged, there will be no messages in this slot, and it will return a value of NULL, e.g.:

model_converged <- is.null(m@optinfo$conv$lme4$messages)

if (model_converged) {
  ...
} else {
  ...
}

4.9 Addressing Nonconvergence

It is worth saying that convergence warnings aren’t always totally reliable. It’s very difficult to estimate whether the nonlinear optimisers used by lme4 have actually converged. Sometimes convergence warnings may be false positives. Even so, a model that passes these convergence tests is much more trustworthy, so we probably want to try to address any issues we see with convergence.

There are a few ways to deal with convergence issues:

  1. Ensure the model is actually estimable
  2. Rescale continuous predictors
  3. Simplify model structure
  4. Change or tweak the optimiser

Find more details in ?lme4::convergence.

4.9.1 Ensure the Model is Estimable

Sometimes models fail to converge because we don’t have enough information in our data to estimate the structure we requested. Addressing this is a matter of thinking through the parameters that the model is estimating, and making sure that each parameter makes sense.

For example, a model that tries to estimate a random slope for the effect of zipf within items is likely to either not converge or produce singular estimates:

bad_m <- lmer(
  log_rt ~ 1 + zipf +
    (1 + zipf | i_id) +
    (1 + zipf | p_id),
  data = trials
)
boundary (singular) fit: see help('isSingular')

In this case, the problem is that (1 + zipf | i_id) is not estimable, since each unique item ID (i_id) has only one value of zipf. This means that we can’t estimate what the response to that item would look like if it had a different zipf value, because this information is missing from the data. The optimiser tries its best and provides a very small estimate for the SD of this random effect.

4.9.2 Rescale Continuous Predictors

A common fix when we have a continuous predictor is to simply rescale it (z-score, so centre and divide by SD). This helps because the nonlinear optimisers used to fit lmer() models can struggle when predictors have vastly different scales. Our Zipf values vary from 1.5 to 7.5, so this could be causing the problem.

Indeed, scaling zipf addresses the convergence issues:

trials <- mutate(trials, zipf_sc = scale(zipf))

m <- lmer(
  log_rt ~ 1 + zipf_sc +
    (1 | i_id) +
    (1 + zipf_sc | p_id),
  data = trials
)

Scaling continuous predictors in this way every time you estimate a model is good practice to ensure that mixed-effects models are stable. An advantage of using the original scale is that you can more meaningfully interpret the coefficients, but:

  1. You can always transform back to the original scale by multiplying coefficients by the SD of the original scale.
  2. If you have multiple predictors, scaling them in this way will provide standardised coefficients, such that you can meaningfully compare their magnitude.

4.9.3 Simplify Model Structure

Using a more sensible scale has already addressed our convergence issues, but in case the model were still not converging we could also simplify the model structure. When models struggle to fit, one common problem is that the model is struggling to estimate the random correlations. As a result, if it’s preventing our model from converging we could simply fit the model without random correlations after all.

m <- lmer(
  log_rt ~ 1 + zipf_sc +
    (1 | i_id) +
    (1 + zipf_sc || p_id),
  data = trials
)

4.9.4 Change/Tweak the Optimiser

Finally, you could try fit the model with a different nonlinear optimiser. As the lme4 documentation recommends, one good option may be lme4::allFit(), which fits the model using all possible optimisers.

This can also be used to check for false positives in the convergence check, because if each model is giving very similar estimates despite using different algorithms, this suggests that the models are in fact converging.
